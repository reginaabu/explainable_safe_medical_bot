## Gold Evidence Definition

### What it is
For every question in PubMedQA `pqa_labeled`, the dataset provides the specific PubMed article (identified by its **PMID / `pubid` field**) that human annotators used as the evidence context. A retrieval result is counted as a **hit** if at least one document chunk from that annotated article appears in the top-k results.

Matching is done by **exact PMID** — no word-overlap heuristics.

### Why it is valid
- PubMedQA was built by linking each question to a real PubMed abstract that contains the answer.
- The PMID is stable and unambiguous — it refers to a specific peer-reviewed article.
- Using PMID matching avoids the threshold-tuning problems of word-overlap proxies.

### Limitations
1. **Single-document gold**: each question has exactly one gold article. There may be other valid evidence articles in PubMed that are not captured.
2. **Chunk-level leniency**: a hit is credited if *any* chunk from the gold article is retrieved. An adversarial evaluator could require the *specific* chunk containing the answer sentence.
3. **In-corpus retrieval only**: the corpus is the 1,000 PubMedQA contexts themselves, not all of PubMed. Scores reflect retrieval within a closed, relatively small corpus.
4. **Dataset construction bias**: because the abstracts were written to answer the questions, lexical overlap between questions and their gold contexts is naturally high — this inflates BM25 recall compared to open-domain retrieval.

## Preprocessing Choices

See [`docs/preprocessing.md`](./docs/preprocessing.md) for full details.

**Summary:**
- Lowercasing: **yes** (at query time and index time)
- Stopwords: **not removed**
- Tokenization: **whitespace split** (no punctuation stripping, no stemming)
- Chunking: **400 words per chunk, 50-word overlap** (step = 350 words)

## Track 2/3 – Knowledge Graph Expansion

### What it does

1. **Entity extraction** — runs SciSpacy NER (`en_ner_bc5cdr_md`: DISEASE + CHEMICAL; fallback: `en_core_sci_sm`: ENTITY) on all 1,000 PubMedQA abstracts → `entities.csv`
2. **KG construction** — builds `triples.csv` with two relation types:
   - `co_occurs_with`: entity pairs that appear in the same abstract (in-memory undirected graph)
   - `mentioned_in`: entity → doc_id provenance links
3. **Query expansion** — `expand_query(query)` in `kg_expand.py` extracts entities from the query, looks up their top co-occurring neighbours in the KG, and appends them to the query string
4. **Evaluation** — compares Recall@5/10, MRR@10, nDCG@10 for BM25-original vs BM25-expanded → `eval_results.md`

### How to run Track 2/3

```bash
# 1. Install SciSpacy and a model (bc5cdr preferred)
pip install scispacy
pip install https://s3-us-west-2.amazonaws.com/ai2-s3-scispacy/releases/v0.5.4/en_ner_bc5cdr_md-0.5.4.tar.gz

# 2. Run the full pipeline (~10–20 min on CPU for NER pass)
python track2_build_kg.py
```

Outputs written to the repo root:
- `pubmedqa_subset.csv` — 1,000 records
- `entities.csv` — all entity mentions
- `triples.csv` — KG edges
- `eval_results.md` — before/after metrics table

### Using `kg_expand.py` as a module

```python
from kg_expand import expand_query, extract_entities, get_entity_neighbors

# Expand a query
q = "Do statins reduce cardiovascular mortality?"
print(expand_query(q))

# Inspect KG neighbourhood
print(get_entity_neighbors("statin", top_n=5))
```

> `triples.csv` must exist before importing (run `track2_build_kg.py` first).

## Debugging

### Enabling debug mode

Append `?debugMode=true` to the app URL — no login or config change needed:

```
http://localhost:8501?debugMode=true
```

The sidebar opens automatically and shows the last 25 lines of `logs/app.log`,
refreshing on every Streamlit rerun. To disable, remove the query parameter or
navigate back to the plain URL.

### What gets logged

Every user interaction writes structured entries to `logs/app.log`:

| Event | When | Key fields |
|---|---|---|
| `QUERY` | Every search | `text`, `top_k`, `mode`, `bm25_candidates`, `top3_pmids`, `top3_norm_scores` |
| `KG_EXPAND` | KG or KG+CE mode | `original`, `expanded`, `terms_added`, `new_terms` |
| `RERANK` | CE or KG+CE mode | `query`, `top3_pmids`, `top3_ce_scores` |
| `RAG_GEN` | Generate Answer click | `query`, `latency`, `answer_words`, `source_pmids` |
| `EVAL` | After each generation | `safe`, `faithfulness`, `relevancy`, `factuality`, `facts` supported/total, `latency` |
| `CORRECTION` (WARNING) | Factuality below threshold | `factuality`, `threshold` |
| `CORRECTION` (INFO) | After retry | `accepted`, `old_factuality`, `new_factuality` |

### Log format

```
YYYY-MM-DD HH:MM:SS | LEVEL    | arogyasaathi | EVENT | field=value …
```

Example entries:

```
2026-02-24 10:12:03 | INFO     | arogyasaathi | QUERY | text='Do statins reduce cardiovascular mortality?' | top_k=5 | mode=KG expansion | bm25_candidates=20 | top3_pmids=['23726098', '17088559', '19289821'] | top3_norm_scores=['1.000', '0.847', '0.821']
2026-02-24 10:12:03 | INFO     | arogyasaathi | KG_EXPAND | original='Do statins reduce cardiovascular mortality?' | expanded='Do statins reduce cardiovascular mortality? LDL cholesterol' | terms_added=2 | new_terms=['LDL', 'cholesterol']
2026-02-24 10:12:15 | INFO     | arogyasaathi | RAG_GEN | query='Do statins reduce cardiovascular mortality?' | latency=11.84s | answer_words=97 | source_pmids=['23726098', '17088559', '19289821']
2026-02-24 10:12:28 | INFO     | arogyasaathi | EVAL | safe=True | faithfulness=0.833 | relevancy=0.761 | factuality=0.60 | facts=3/5 | latency=13.20s
```

### Reading logs from the terminal

```bash
# Follow the log in real time
tail -f logs/app.log

# Filter to a specific event type
grep "RAG_GEN"   logs/app.log
grep "EVAL"      logs/app.log
grep "CORRECTION" logs/app.log

# Show all queries run today
grep "QUERY" logs/app.log | grep "$(date +%Y-%m-%d)"
```
